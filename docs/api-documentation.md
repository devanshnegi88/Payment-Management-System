# API Documentation

Base URL (local dev): `http://127.0.0.1:8000`

Interactive Swagger docs are also auto-generated by FastAPI at `/docs`,
and ReDoc at `/redoc`.

---

## Sales

### `POST /sales`
Creates a new sale (always starts as `pending`). Auto-creates the user if
`user_id` doesn't exist yet.

**Request**
```json
{ "user_id": "john_doe", "brand": "brand_1", "earning": 40 }
```

**Response** `201 Created`
```json
{
  "id": 1,
  "user_id": "john_doe",
  "brand": "brand_1",
  "status": "pending",
  "earning": 40.0,
  "advance_paid": 0.0,
  "is_reconciled": false,
  "created_at": "2026-07-19T10:00:00Z",
  "reconciled_at": null
}
```

### `GET /sales/{sale_id}`
Fetches one sale by ID. `404` if not found.

### `GET /sales?user_id=john_doe`
Lists all sales for a user. `404` if the user doesn't exist.

### `PATCH /sales/{sale_id}/reconcile`
Admin action: reconciles a single sale to `approved` or `rejected`,
applying the final payout adjustment.

**Request**
```json
{ "status": "approved" }
```

**Response** `200 OK` — updated `SaleResponse`.

**Errors**
- `404` — sale not found
- `400` — sale is not `pending`, already reconciled, or invalid target status

### `POST /sales/reconcile/batch`
Admin action: reconciles multiple sales for one user in a single call,
returning the aggregated final payout — mirrors the assignment's worked
example.

**Request**
```json
{
  "user_id": "john_doe",
  "reconciliations": [
    { "sale_id": 1, "status": "rejected" },
    { "sale_id": 2, "status": "approved" },
    { "sale_id": 3, "status": "approved" }
  ]
}
```

**Response** `200 OK`
```json
{
  "user_id": "john_doe",
  "total_final_payout": 68.0,
  "breakdown": [
    { "sale_id": 1, "status": "rejected", "earning": 40.0, "advance_paid": 4.0, "adjustment": -4.0 },
    { "sale_id": 2, "status": "approved", "earning": 40.0, "advance_paid": 4.0, "adjustment": 36.0 },
    { "sale_id": 3, "status": "approved", "earning": 40.0, "advance_paid": 4.0, "adjustment": 36.0 }
  ]
}
```

## Payouts

### `POST /payouts/advance/{user_id}`
Runs the advance payout job for one user. **Safe to call repeatedly** —
already-advanced sales are skipped, not re-paid.

**Response** `200 OK`
```json
{
  "user_id": "john_doe",
  "total_advance_paid": 12.0,
  "sales_paid": [1, 2, 3],
  "sales_skipped": []
}
```

### `POST /payouts/advance`
Runs the advance payout job across every user (simulates a scheduled
batch job). Returns a list of `AdvancePayoutResult`.

### `GET /payouts?user_id=john_doe`
Returns the full payout ledger (advance, final, withdrawal entries) for a
user.

### `PATCH /payouts/{payout_id}/status`
Simulates a payment processor callback updating a payout's status. If the
new status is `failed`, `cancelled`, or `rejected`, the amount is
**automatically credited back** to the user's withdrawable balance
(Question 2).

**Request**
```json
{ "status": "failed" }
```

**Response** `200 OK` — updated `PayoutResponse`, with `recovered_at` set.

### `POST /payouts/recover/{user_id}`
Defensive batch sweep: recovers any of the user's bad-state payouts that
were never credited back (e.g. a missed webhook). Returns the list of
payouts that were recovered in this call (empty if none were pending
recovery).

## Withdrawals

### `POST /withdrawals/{user_id}`
Withdraws the user's full `withdrawable_balance`, subject to the 24-hour
cooldown.

**Response** `200 OK`
```json
{
  "user_id": "john_doe",
  "amount_withdrawn": 68.0,
  "withdrawn_at": "2026-07-19T10:05:00Z",
  "next_withdrawal_available_at": "2026-07-20T10:05:00Z"
}
```

**Errors**
- `404` — user not found
- `429 Too Many Requests` — cooldown not yet elapsed (message includes when
  the next withdrawal becomes available)
- `400` — balance is zero, nothing to withdraw

## Example End-to-End Flow

```bash
# 1. Create 3 pending sales of ₹40 each for john_doe
curl -X POST /sales -d '{"user_id":"john_doe","brand":"brand_1","earning":40}'
curl -X POST /sales -d '{"user_id":"john_doe","brand":"brand_1","earning":40}'
curl -X POST /sales -d '{"user_id":"john_doe","brand":"brand_1","earning":40}'

# 2. Run the advance payout job -> pays ₹12 total (₹4 per sale)
curl -X POST /payouts/advance/john_doe

# 3. Admin reconciles: 1 rejected, 2 approved -> final payout ₹68
curl -X POST /sales/reconcile/batch -d '{
  "user_id": "john_doe",
  "reconciliations": [
    {"sale_id": 1, "status": "rejected"},
    {"sale_id": 2, "status": "approved"},
    {"sale_id": 3, "status": "approved"}
  ]
}'

# 4. User withdraws ₹80 total (₹12 advance + ₹68 final = ₹80)
curl -X POST /withdrawals/john_doe

# 5. Suppose that withdrawal payout later fails at the bank
curl -X PATCH /payouts/{payout_id}/status -d '{"status":"failed"}'
# -> withdrawable_balance is credited back to ₹80 immediately

# 6. The user CAN initiate another withdrawal for that amount, but note:
#    the 24h cooldown timer was already started by the original withdrawal
#    in step 4 and is NOT reset by recovery. If less than 24h have passed,
#    this call still returns 429 until the cooldown from step 4 elapses.
#    See docs/edge-cases.md ("Recovery vs. cooldown interaction") for the
#    reasoning behind this design choice.
curl -X POST /withdrawals/john_doe
```
